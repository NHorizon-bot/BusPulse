# 网约公交智能诊断 Agent

> **文档状态**：调研稿

## Executive Summary

### 问题：网约公交客诉诊断为何是"吞人力"黑洞？

网约公交（需求响应式公交 / 动态拼车 / 场站调度）的高频客诉诊断，长期依赖 Oncall 研发和高级运营在 **4~6 个系统间手动切换**：登 Grafana 看 Trace → 进 Kibana 捞算法日志 → 登录运营后台查围栏 → 查高德路况 → 翻司机 GPS 轨迹。**单案平均诊断时长（MTTD）5~15 分钟**，且 80% 的重复性 Case（围栏超区、站点失效、算法策略拦截）消耗了同等的技术人力。

**核心矛盾**：一线客诉量在增长，但具备"看懂算法日志 + 理解 RPC 堆栈"能力的复合型排查人力无法同比扩充。

### 方案：Eino ADK Agent + 只读工具链

本报告设计的**网约公交智能诊断 Agent**，核心不是聊天机器人，而是一个具备自省能力、多源异步并发取数、严格遵循排障 SOP 的「推理-行动」闭环系统：

| 维度 | 方案选型 | 选型原因 |
|------|---------|---------|
| **后端语言** | Go | 支撑千万级并发 |
| **AI 框架** | Eino（CloudWeGo 开源 Go LLM 框架，https://github.com/cloudwego/eino） | Go 原生框架，无 Python 桥接安全与性能损耗。内置 ChatModelAgent 封装 ReAct 循环、Tool Calling、Stream 处理，无需手写状态机 |
| **控制架构** | Eino ADK Agent + Compose Graph 编排 | ChatModelAgent 内置 ReAct 循环（MaxIterations 配置即熔断）；规则引擎分流由 Compose Graph 的 Branch 机制声明式实现，比手写 if-else 更易维护 |
| **知识底座** | 规则引擎 → LLM 双路 RAG（Eino-ext Retriever 抽象） | 规则引擎兜底 60% 确定性 Case，零 Token 成本；LLM 卡位 40% 长尾策略异常。RAG 层通过 Eino-ext Retriever 与向量库/ES 解耦，冷启动至 P2 阶段可无缝切换存储后端 |
| **交互渠道** | Link机器人卡片 + 工单系统微插件 | 零重构嵌入现有流程，客服无需跳出系统 |

### 核心目标

> **MTTD**：5 分钟 → **10 秒内**（提效 **95%**）  
> **人工深度排查率**：当前 100% → **20%~30%**（下降 **70%~80%**）  
> **单人日均处理量**：每人 15~20 单/日 → **60~80 单/日**（提升 **3~4 倍**）

## 二、后台原子能力矩阵

### 2.0 Agent 核心诊断工作流

Agent 的每笔诊断执行 **6 步闭环**。Step ③↔④ 构成 ReAct 推理循环，最大 4 轮后硬性熔断，确保 3~5 秒内收敛到根因。

```
[ 1. 接收输入与意图识别 ]
   （订单号 / 报警事件 → 提取 order_id, city_id, 故障大类）
           │
           ▼
[ 2. 制定初始排查策略 ] ◄─── 结合【业务拓扑 + SOP 知识库】
   （根据故障类型，编排首轮工具调用顺序与优先级）
           │
           ▼
┌──► [ 3. 执行工具调用 (Action) ]
│      （并发调 5~8 个只读 RPC，1s 内返回脱敏摘要）
│           │
│           ▼
└─── [ 4. 观察与分析 (Observation) ]
      （LLM 检查返回数据 → 是否定位根因 → 否，回 Step 3 补充调用）
               │
               ▼ (已捕获根因特征)
      [ 5. 根因整合与大白话翻译 ]
      （技术日志 → 业务大白话 + 安抚话术）
               │
               ▼
      [ 6. 给出修复建议 / 联动触发 ]
      （自动工单 / 建议话术 / 一键回滚 / 补偿）
```

> Step ③↔④ 循环的熔断保护：每轮消耗约 1~2K Token，4 轮累计 ≤ 8K Token。
> 规则引擎在 Step ① 命中的场景直接跳过循环，零 Token 成本输出结论。

### 2.2 Go 并发工具链（核心性能底座）

Agent 后台在 1 秒内并发调取 5~8 个只读 RPC。Eino 自身也是 Go 应用，工具层直接在 LambdaNode 中编写标准 Go 并发代码即可：

```go
// 诊断入口：1.5s 全局硬截止
diagCtx, cancel := context.WithTimeout(ctx, 1500*time.Millisecond)
defer cancel()

// errgroup 扇出-汇聚
g, diagCtx := errgroup.WithContext(diagCtx)
g.Go(func() error { return fetchOrderContext(diagCtx, orderId) })  // 超时500ms
g.Go(func() error { return fetchVehicleGPS(diagCtx, vehicleId) })  // 超时800ms
g.Go(func() error { return fetchAlgoSnapshot(diagCtx, orderId) })  // 超时1500ms
g.Go(func() error { return fetchTraceLog(diagCtx, traceId) })      // 超时1000ms
```

| 技术机制 | 作用 | 手写 Go vs Eino |
|---------|------|-----------------|
| **Goroutine + errgroup** | 扇出并发调多路工具，首个 error 取消全体 | 手写与 Eino 均可直接使用 Go 标准库，无差别 |
| **context.WithTimeout** | 1.5s 全局硬截止，防止下游卡死拖垮服务 | Eino Graph 节点可独立持有 Context，更细粒度超时 |
| **Tool 注册与路由** | LLM 输出的 tool_call 分派到对应工具 | 手写需 switch-case dispatch map；Eino 框架自动路由到 tool.BaseTool 实现 |
| **流式渐进渲染** | 1s 骨架 → 3~5s 深度诊断分阶段流式输出 | 手写需自行管理 stream；Eino Compose 支持流式编排 |
| **审计日志注入** | 每笔诊断全链路埋点 | 手写每个工具内加日志；Eino Callback Aspects 统一注入 |
| **Human-in-the-Loop** | 人工复核暂停 / 恢复诊断 | 手写需 checkpoint 持久化中间状态；Eino 内置 Interrupt/Resume |
| **并发控制 ≤10** | 防止雪崩 | 均可使用 semaphore.Weighted，Eino 不限制

### 2.3 脱敏与摘要网关

大模型不能直接吞下数万行原始日志。工具层在返回数据前执行**两步剪裁**：

1. **敏感字段擦除**：过滤手机号、身份证、经纬度精确值（→ 网格化模糊）
2. **核心特征摘要**：从算法匹配矩阵日志中，通过 Go 正则只保留含 `FILTER`、`TIMEOUT`、`MISMATCH`、`LIMIT_EXCEEDED` 的关键行，合并为 ≤ 1KB 的精简 JSON

### 2.4 规则引擎 + LLM 双路 RAG

```
  [ 故障输入 ]
        │
        ▼
┌───────────────────────────────┐
│  规则引擎诊断（前置）            │
│  - 状态码 404/500/504 → 秒杀   │
│  - GPS 出圈 → 秒杀             │
│  - 站点 status=0 → 秒杀        │
└───────────┬───────────────────┘
            │
   ┌────────┴────────────┐
   ▼                     ▼
[规则已覆盖]          [规则无法解释]
    │                     │
    ▼                     ▼
直接输出确定性结论    激活 LLM Agent
    │               ┌──────────┐
    │               │ RAG 双路 │
    │               │ 1. SOP 向量库
    │               │ 2. 历史根因库
    │               │ 3. 脱敏后工具结果
    │               └────┬─────┘
    │                    ▼
    │               LLM 深度推理
    │               (Max 4步，硬熔断)
    │                    │
    └──────┬─────────────┘
           ▼
    [结构化诊断报告]
```

> **冷启动策略**：规则引擎优先上马，立即可覆盖 60% 基础 Case（零 Token 成本、零幻觉）。LLM 仅卡位 40% 跨系统长尾异常。新发现的固定模式反哺沉淀为规则，形成"规则盘越滚越大"的飞轮。
>
> **RAG 存储解耦**：SOP 向量库通过 Eino-ext Retriever 接口接入，无论用 Milvus、ES dense_vector 还是 pgvector，上层 Agent 代码不变。冷启动阶段 SOP 量级 < 10 条，直接注入 Agent Instruction（FString 模板）比向量检索更快更准，向量库在 P2 阶段接入即可。

## 三、核心业务提效指标

### 3.1 时效指标

| 指标 | 现状（人工） | 目标（Agent） | 提效增幅 |
|------|------------|-------------|---------|
| **单案平均诊断时长（MTTD）** | 5~15 分钟 | ≤ 10 秒（核心推理 3~5 秒） | **↑ 95%** |
| **多系统切换次数** | 4~6 次（Grafana / Kibana / 运营后台 / 高德地图） | 0 次（Agent 后台一次性并发聚合） | **取消切换** |
| **首屏渲染响应** | 无（需等全部查完） | 1 秒内闪电渲染时空骨架 | **即时反馈** |

### 3.2 效能与介入率指标

| 指标 | 现状 | 目标 | 说明 |
|------|------|------|------|
| **人工深度排查率** | 100%（每个 Case 都要看日志） | 20%~30%（仅极度复杂 Case 转人工） | **↓ 70%~80%** |
| **单人日均处理量** | 15~20 单/人/日 | 60~80 单/人/日 | **↑ 3~4 倍** |
| **单案操作时间** | 3 分钟（查日志→定位→写话术→提单） | 10 秒（看一眼报告→点击确认） | **↓ 95%** |

### 3.3 质量指标 & 评测方法

**评测基线**：MVP 开工第一周即用 **Eino-ext Devops Eval** 建立 30~50 条黄金测试 Case（要求运营配合梳理）。每次 Prompt 迭代、模型切换、规则新增均触发全量评测流水线，定量对比准确率变化，杜绝"靠感觉判断质量"。

| 指标 | 目标 | 验收标准 | 评测方式 |
|------|------|---------|---------|
| **诊断准确率（MVP）** | ≥ 70% | 人工复核确认率 ≥ 70% | Eino-ext Eval 批量跑 30~50 条黄金 Case，对比诊断结论与标注答案 |
| **诊断准确率（稳态）** | ≥ 90% | 数据循环运行 2 个月后达 90%+ | Eino-ext Eval 持续积累 Case 库至 200+ 条，每周自动回归 |
| **诊断召回率** | ≥ 95% | 90%+ 高频 Case 类型可被自动诊断覆盖 | Eino-ext Eval 按场景维度分组统计已覆盖 vs 未覆盖 |
| **误诊率** | ≤ 5% | 错误根因导致错误 Action 的比例 | 人工复核的拒绝率 + Eval 误判分析 |

## 四、权限与审计合规方案

### 4.1 权限最小化原则

| 安全维度 | 方案 | 控制粒度 |
|---------|------|---------|
| **工具权限** | 所有底层微服务接口注册为 **Read-Only** | 接口级，禁止任何写操作 |
| **数据脱敏** | 手机号/身份证 → SHA256 加盐混淆；精确 GPS → 网格化模糊 | 字段级 |
| **写操作卡点** | Agent 不执行写操作，仅生成 Action 建议；人工点击确认后才通过独立通道执行 | 操作级 |
| **模型安全** | 脱敏后数据才进入 LLM Context；调用外部 API 时经过安全网关 | 通道级 |

### 4.2 诊断可观测性 & 全日志留痕

每笔诊断生成审计记录的同时，建立三层可观测性体系，覆盖审计合规与生产排查的双重需求：

**第一层 — Eino Callback（框架级，零代码注入）**

Eino 的 Callback Aspects（OnStart/OnEnd/OnError）在框架层面自动捕获 Agent 全生命周期事件，无需在每个工具中手动埋点：

| 捕获内容 | 示例值 |
|---------|--------|
| 每个 Node 的启动/结束/错误 | `rule_engine OnEnd: duration=3ms, err=nil` |
| LLM 调用详情 | `model=deepseek-chat, prompt_tokens=1245, completion_tokens=356` |
| Tool 调用详情 | `GetOrderContext: status=SUCCESS, duration=120ms` |
| 流式输出阶段 | `skeleton output → deep_diagnose output` |

**第二层 — 结构化审计日志（持久化，支持审计回溯）**

```json
{
  "audit_id": "aud_20260709_xyz888",
  "operator_id": "xiaoming_v",
  "trigger_source": "FEISHU_CARD | WIDGET",
  "order_id": "88992314",
  "timestamp": 1783584000,
  "agent_tools_invoked": [
    {"tool": "GetOrderContext", "duration_ms": 120, "status": "SUCCESS"},
    {"tool": "GetAlgoSnapshot", "duration_ms": 450, "status": "SUCCESS"}
  ],
  "llm_raw_response_digest": "sha256:abc123...",
  "final_diagnostic_json": { "...": "..." },
  "human_action": "CONFIRMED | REJECTED",
  "human_correction_note": "驳回后人工填写的纠偏备注"
}
```

**第三层 — OpenTelemetry 全链路追踪（P2 阶段接入，生产排查用）**

MVP 阶段 Callback 日志够用。进入生产态后接入 OpenTelemetry，回答审计日志回答不了的问题：

| 问题场景 | 审计日志 | + OpenTelemetry |
|---------|---------|----------------|
| "哪个工具慢了？" | ✅ 能答 (duration_ms) | ✅ 更细 (网络/模型推理/Tool 内各子步骤) |
| "Agent 为什么得出这个结论？" | ❌ 只存最终 JSON | ✅ 全量回放：Thought→ToolCall→Observation 每步留痕 |
| "换了模型/改 Prompt 后效果怎么变？" | ❌ 无对比 | ✅ 按 model_name/prompt_version 聚合仪表盘 |
| "跨服务链路哪里超时了？" | ❌ 只看到本 Agent | ✅ Trace ID 关联下游 RPC/DB 调用 |

### 4.3 人工复核闭环（Human-in-the-Loop）

```
Agent 产出诊断报告 ──► 人工复核
                           │
              ┌────────────┴────────────┐
              ▼                         ▼
     [ 🟢 确认结论 ]            [ 🔴 驳回转人工 ]
              │                         │
              ▼                         ▼
    自动执行推荐 Action         人工填写正确根因 + 话术
              │                         │
              ▼                         ▼
    进入正样本 RAG 库           负样本 → Prompt 迭代 / LoRA 微调
```

## 五、MVP 阶段落地排查字典（9 大高频客诉场景）

### 5.1 重点展开：3 个最高 ROI 场景

#### 场景一：乘客端看不到网约公交 / 无法发单（金刚位无入口）

**痛点等级**：★★★★★（最高频准入类客诉）

| 维度 | 内容 |
|------|------|
| **诊断工具链** | `GetAppConfig(user_id, city_id)` → `GetGeoFence(lng, lat)` |
| **规则引擎判定** | geo_fence 匹配 → Status 命中 → 秒级结论 |
| **底层根因** | ① 城市/站点运营时段未开启（22:00 后下线）② GPS 处于开通城市围栏外 ③ 营销灰度 A/B 排除 |
| **Agent 话术** | "非常抱歉，网约公交目前处于固定时段（早7点至晚9点）运营，该服务仅在限定区域内开通，您当前的位置或叫车时间暂时超出了服务范围。" |
| **推荐动作** | 告知运营当前运营时段 & 服务区范围 |

#### 场景二：算法策略拦截导致"有车不派 / 搜不到线"

**痛点等级**：★★★★★（排查最耗时、技术门槛最高）

| 维度 | 内容 |
|------|------|
| **诊断工具链** | `GetAlgoSnapshot(order_id)` → 扫描 Filter_Matrix |
| **规则引擎判定** | 命中 `Heading_Mismatch` 或 `Delay_Time_Exceeded` |
| **底层根因** | ① **方向冲突**：周边车辆的行驶方向与乘客目的地反向，掉头时间 > 直行时间 ② **在车旅客体验保护**：若插入该拼车单，导致在车旅客 ETA 延误 > 15 分钟阈值 |
| **Agent 话术** | "网约公交是全盘动态匹配的。距离您近的车辆行驶方向与您相反，或者它已经被算法分配给了更顺路的拼车队列，系统为您选择的是综合预计到达时间最快的车辆。" |
| **推荐动作** | 告知算法配置阈值，无需人工干预 |

#### 场景三：技术链路超时 / 大盘报错（500/504）

**痛点等级**：★★★★☆（批量进线必杀场景）

| 维度 | 内容 |
|------|------|
| **诊断工具链** | `GetTraceLog(trace_id, span_id)` → Jaeger 下钻至最底层 Span |
| **规则引擎判定** | HTTP 5xx 或 RPC Timeout → 直接触发 Trace 诊断 |
| **底层根因** | 晚高峰流量飙升 → 下游微服务连接池耗尽 / MySQL 慢查询 → 接口超时 |
| **Agent 话术** | （面向研发）"Trace ID tr_xxx 显示 bus-user-profile 微服务 Redis 连接池耗尽，建议扩容或优化连接配置。" |
| **推荐动作** | 自动生成 Jira，标明 Trace ID 和根因 Span |

### 5.2 全量 9 大客诉诊断排查字典

| ID | 高频客诉痛点 | 工具链 | 规则引擎 | 底层根因 | 话术（面向用户） |
|----|------------|--------|---------|---------|----------------|
| 1 | **金刚位无入口** | GetAppConfig + GetGeoFence | ✅ GPS 出圈 / 停运时段 | 运营区域未覆盖 或 灰度排除 | "当前超出服务范围" |
| 2 | **反作弊恶意取消拦截** | GetUserTag + GetCancelHistory | ✅ 24h 取消 ≥ 3 次 | 反作弊防占座策略命中 | "检测到频繁取消，请过段时间再试" |
| 3 | **司机到站无法流转** | GetDriverAppStatus + GetGpsDistance | ✅ GPS 距站点 > 100m | 防早击 GPS 锁限制 或 App 切后台 | "请驶入站点并保持 App 前台运行" |
| 4 | **补票余座锁死** | GetLineMatch + GetVehicleCapacity | ✅ LockedSeats ≥ TotalSeats | 后续站点线上锁座已满 | "座位已被后续站点预约锁定" |

| 8 | **大车导航限行** | GetMapRoute + GetTrafficStatus | ✅ 限高/限宽路由 | 公交车型限行或深红拥堵避堵 | "大巴自动避开限高路段" |
| 9 | **接驾超阈值** | GetEtaSnapshot(order_id) | ✅ 周边无顺路车 | 时空断层跨区拉车 或 前序乘客迟到 | "周边运力紧张，调度了稍远车辆" |

### 5.3 规则清单（确定性秒杀规则，零 Token 成本）

| 规则 ID | 场景 | 触发特征 | 判定逻辑 | 结论 |
|---------|------|---------|---------|------|
| R-01 | 下单失败 | 坐标不在 GeoFence 内部 | 经纬度多边形碰撞检测 | 超区下单 |
| R-02 | 下单失败 | 500m 内无活跃虚拟站点 | Station_Status 字典 Status=0 | 虚拟站点失效 |

| R-04 | 派单失败 | Filter_Matrix 命中 Heading_Mismatch | 方向夹角 > 90° | 时空方向冲突 |
| R-05 | 派单失败 | ΔETA > 15min | 在车旅客延误保护 | 体验保护触发 |
| R-06 | 接驾迟到 | GPS 偏离推荐路线 > 500m 且持续 > 3min | 轨迹偏离度计算 | 司机主观走错路 |
| R-07 | 接驾迟到 | Congestion_Index ≥ 3.5（深红） | 路况 API 查询 | 客观突发拥堵 |

| R-10 | 系统报错 | 504 + Redis 连接池耗尽 | Trace 下钻定位 | 下游连接池耗尽 |

## 六、数据循环 & 质量飞轮

### 6.1 评测基线先行

MVP 开工第一周即建立评测基线，避免"凭感觉判断质量"：

| 步骤 | 做法 | 工具 |
|------|------|------|
| ① 建黄金 Case 库 | 运营梳理 30~50 条典型客诉 + 标注正确诊断结论 | 运营人工 |
| ② 首轮评测 | 跑 Agent vs 标注结论，算准确率/召回率 | Eino-ext Devops Eval |
| ③ 迭代优化 | 根据失败 Case 修 Prompt / 加规则 → 再评测 | Eino-ext Devops Eval |
| ④ 沉淀为回归 | 每次 Prompt 改版 / 模型切换 / 规则增删，全量跑一遍 | Eino-ext Devops Eval 自动化 |

### 6.2 持续飞轮

```
                               ┌──────────────────┐
                               │  Eval 评测基线     │
                               │  (30→50→200+ Case)│
                               └────────┬─────────┘
                                        │
人工确认结论 ──► 正样本 ──► RAG向量库 ──┼──► 相似Case自动命中
                                        │
人工驳回修正 ──► 负样本 ──► Prompt迭代 ─┼──► Eval 跑分 → 是否提升？
                                        │
                               ▼        ▼
                    准确率 ≥ 90% ?   准确率未达标
                          │              │
                          ▼              ▼
                   固化到规则引擎     继续迭代 Prompt
                    (零Token成本)      (回归保护防止劣化)
```

| 里程碑 | 状态 | 评测基线规模 | 预期准确率 |
|--------|------|------------|-----------|
| P0 基建（T+1 周） | 规则引擎 + Eval 基线搭建 | 30 条 | 规则场景 100%（确定性） |
| P1 核心场景（T+3 周） | Agent 推理链 + 首轮评测 | 50 条 | ≥ 70% |
| P2 集成（T+5 周） | 全量打流 + 自动回归 | 100 条 | ≥ 80% |
| P3 稳态（T+6 周+） | 飞轮自动运转 | 200+ 条 | ≥ 90% |
